Section I: Bezier Curves and Surfaces

Part 1: Bezier curves with 1D de Casteljau subdivision

Casteljau’s algorithm is a recursive method that can be used to evaluate a point on a Bezier curve for a given parameter t (where t ranges from 0 to 1) using linear interpolation instead of directly calculating each Bernstein polynomial. The logic involves taking a given n control points on a Bezier curve and finding the weighted average of each of the adjacent points to create a new set of n - 1 points (recursive step). This process will continue until there is only 1 point left (base case). This final point is the position on the Bezier curve for the given value t.

When we implemented BezierCurve::evaluateStep, our main goal is to recreate one step of the recursive call in Casteljau’s algorithm: returning the n - 1 new points. This logic involved iterating through the given vector points which contains n points to produce n - 1 new points. In order to find our new intermediate points, we used linear interpolation on points[i] and points[i + 1] or the formula P = (1 - t) * Pi + t * Pi+1. Then when we finish doing this for all points in vector points, we return all the new n - 1 intermediate points in a new vector.

Basically, evaluateStep completes one step of subdivision, so when we’re in the meshedit GUI we need to repeatedly press E (n - 1 times) in order to complete each step of Casteljau’s algorithm and get our final point, as demonstrated in this example with a 6 point Bezier curve:

Part 2: Bezier surfaces with separable 1D de Casteljau

Since a Bezier surface is defined by an n * n grid of control points and two parameters, u and v, we can treat this grid as a series of independent Bezier curves and extend Casteljau’s algorithm to Bezier surfaces. This can be accomplished by reducing each of the n rows of control points to a single using the parameter u, creating a new “moving curve.” Then we can evaluate this curve with parameter v to find our (u, v) on the Bezier surface.

In our code implementation, we created 3 functions. The first was BezierPatch::evaluateStep which was the same as BezierCurve::evaluateStep except we used 3D points instead of 2D points, as we are now working with a 3D surface instead of a 2D curve. Next, we implemented BezierPatch::evaluate1D which is the step that creates our “moving curve.” This function works by taking points and a parameter t and calls evaluateStep until there is only 1 point left. Finally, we implemented BezierPatch::evaluate which takes our n * n surface and separates it into rows, calling evaluate1D on each row with parameter u. Then evaluate1D is called one more time on the vector that holds all the evaluated points from each row and parameter v to return the final 3D point on the surface.

Section II: Triangle Meshes and Half-Edge Data Structure

Part 3: Area-weighted vertex normals

In order to implement area-weighted vertex normals for smooth shading, we utilized the half-edge data structure to iterate through all faces based on a given vertex, starting with its primary half-edge. In order to traverse the half-edge, we used a do-while loop and traversed to the next half-edge using curr->twin()->next(), stopping when the loop gets back to the starting point. In order to handle boundaries (ie. holes or mesh edges), we added a condition !curr->face()->isBoundary() to ensure we are not trying to include boundary faces in our calculations. Next, for each triangle face, we used its three vertices (p0, p1, p2) with p0 being the central vertex, and calculated the two edge vectors using e1 = p1 - p0 and e2 = p2 - p0. Then we calculated the cross-product of e1 and e2 in order to get a vector pointing perpendicular to its face with a magnitude proportional to the triangle’s area (area-weighted normal). Finally, we summed up the cross-products and used unit() to normalize this sum to return the vector needed for smooth shading.

Part 4: Edge flip

In order to not lose any of the mesh in memory while doing edge-flips, we made sure to collect pointers first and reassign them after all other operations. To start, we added a condition to check if the selected edge was a boundary with isBoundary() because you can’t flip a boundary edge. Next, before actually flipping any of the edges, we thought it would be a good idea to store the area around the edge we want to flip to make sure we don’t lose anything. As a result, we stored the 6 inner half-edges as h_start, h_opp, h_next1, h_next2, h_opp_next1, and h_opp_next2; the 2 adjacent faces as face1 and face2; and 4 vertices as vert_a, vert_b, vert_c, and vert_d that form a quadrilateral around the pair of triangles that share the edge we want to flip. Now that the current state of the mesh is saved, we start reassigning neighbors using the setNeighbors() function with the goal of breaking two circular loops of 3 half-edges to make two new circular loops. For our core flip, we want to flip h_start and h_opp which are the two half-edges that originally shared e0. They originally connected vertices c and b, but we are trying to connect vertices a and d. After making the core flip, we build our two new faces. For face1, we change h_next2 and h_opp_next1 such that they form a triangle with h_start and for face2, we change h_opp_next2 and h_next1 such that they form a triangle with h_opp. Finally, we can update the half-edge pointers using halfedge() for the 4 vertices, 2 faces, and flipped edge.

Part 5: Edge split

To implement edge-splitting while also ensuring the mesh remained completely manifold, we did some pointer resetting while also maintaining old vertices and halfedges.

First, we gathered the existing mesh elements (vertices, edges, faces, and halfedges) of the two triangles.

Then we calculated the midpoint of the target edge and allocated the new elements we needed: 1 new vertex m , 3 new edges (old edge is shrunk to half the size), 2 new faces, and 6 new halfedges. The original edge (+ its halfedges + faces) were repurposed.

Next, setNeighbors()is called on all 12 internal halfedges. We ensured h1, h2, h4, h5 maintained their original vertices and twins to preserve the mesh's connectivity with edges and faces outside of the splitting operation.

Finally, halfedge pointers for the involved vertices, edges and faces were updated and double checked so that they point to valid halfedges.

Explicitly naming all halfedges made it much easier to keep track of elements rather than doing pointers to pointers like next()->twin()->next().

Part 6: Loop subdivision for mesh upsampling

Loop subdivision was implemented while keeping old positions and new posistions in mind so as to not run operations on new positions while iterating through the mesh.

First, we iterated through all original vertices and calculated their smoothed positions using the loop weighting rule from the instructions. The result was stored in Vertex::newPosition and then marked isNew = false .

Next, we iterated through all original edges, calculated the midpoint vertex using the 3/8 and 1/8 rule and stored that in Edge::newPosition.

Then, we iterated through the edges and split them. To avoid infinitely looping, we only processed the original edges. For every split, we assigned the pre-calculated e->newPosition to the newly made vertex and set its isNew to true. Then, we iterated through the edges and flipped any new edges, connecting one old vertex with a new vertex using a xor condition.

Finally, we iterated through all the vertices and copied their newPositions into their positions.

As a note, we had to go back to our splitEdge function from part 5 to label our isNew flags for new and old edges.

After loop division, the overall surface of the object becomes smoother (like rounder). Sharp corners and edges that stick out become rounded off. Anything sticking out are pulled inwards, causing the mesh to sort of shrink or compress inwards.

The effect can be reduced by pre-splitting edges since pre-splitting edges near a corner adds more geometry and pulls the neighbor vertices closer to the corner. Then, when weighted average is calculated, the corner is pulled towards the neighbors meaning it doesn't collapse inwards and maintains the geometry.



After several iterations of loop subdivision, the cube becomes lopsided and asymmetric. This is because it's made of two triangles that are split by a single diagonal edge. Different corners have different degrees/# of edges connecting them. Because the formula with 3/8 and 3/16 depends on the vertex degree, vertices become pulled inward at different rates.



To make it subdivide symmetrically, we need to balance the vertex degrees across all corners. By splitting the diagonal edge on all 6 faces of the cube, each face has an X on it and now each face is composed of 4 symmetric triangles. Splitting the diagonals means each corner has the same degree, making the cube symmetric. This makes it symmetrical.
